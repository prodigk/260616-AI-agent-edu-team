# LLM 답변 워크플로 패턴

참고 문서: Anthropic, "Building effective agents"
https://www.anthropic.com/engineering/building-effective-agents

## 기본 원칙

사용자 질문은 먼저 가장 단순한 방식으로 답변한다. 복잡한 워크플로는 정확도, 검증 가능성, 도구 사용, 다중 관점이 실제로 필요할 때만 선택한다.

모든 답변 하단에는 사용한 워크플로 타입을 반드시 표시한다.

```text
---
답변 방식: <workflow type>
선택 이유: <짧은 이유>
```

아래 5가지에 맞지 않으면 `별도 패턴: <패턴명>`으로 표시한다.

## 1. Augmented LLM

### 정의

단일 LLM 호출에 검색, 크롤링 결과, DB, 파일, 메모리, 계산 도구 같은 외부 컨텍스트를 붙여 답변한다.

### 사용 조건

- 질문이 하나의 답변으로 충분하다.
- 최신 정보, 문서, 기사, 내부 데이터 같은 근거가 필요하다.
- 복잡한 단계 분해나 다중 에이전트가 필요하지 않다.

### 예시

- "인공지능 관련 뉴스 요약해줘"
- "이 URL의 핵심 내용을 요약해줘"
- "스크랩한 기사 목록을 바탕으로 트렌드를 알려줘"

### 답변 패턴

```text
1. 사용자 질문을 이해한다.
2. 필요한 외부 컨텍스트를 가져온다.
3. 컨텍스트에 근거해 답변한다.
4. 불확실하거나 근거가 부족한 부분은 명확히 표시한다.
5. 하단에 답변 방식을 표시한다.
```

### 하단 표시 예

```text
---
답변 방식: Augmented LLM
선택 이유: 크롤링된 기사 목록을 외부 컨텍스트로 사용해 단일 답변을 생성했습니다.
```

## 2. Prompt chaining

### 정의

질문을 고정된 순서의 여러 단계로 나누고, 앞 단계의 결과를 다음 단계 입력으로 사용한다.

### 사용 조건

- 작업 순서가 명확하다.
- 중간 결과를 검사하거나 보정할 필요가 있다.
- 한 번에 답하면 품질이 낮아질 가능성이 크다.

### 예시

- "뉴스 50개를 수집하고, 중복 제거 후, 주제별로 요약해줘"
- "기사 제목을 먼저 분류하고, 각 분류별 핵심 이슈를 작성해줘"
- "초안을 만들고 톤을 바꿔서 최종본을 작성해줘"

### 답변 패턴

```text
1. 단계 1: 입력 또는 자료를 정리한다.
2. 단계 2: 중간 산출물을 만든다.
3. 단계 3: 중간 산출물의 품질을 검사한다.
4. 단계 4: 최종 답변을 작성한다.
5. 하단에 답변 방식을 표시한다.
```

### 하단 표시 예

```text
---
답변 방식: Prompt chaining
선택 이유: 수집, 정제, 분류, 요약이 순차적으로 의존하는 작업이어서 단계별로 처리했습니다.
```

## 3. Routing

### 정의

사용자 질문을 먼저 유형별로 분류한 뒤, 각 유형에 맞는 전문 프롬프트나 처리 경로로 보낸다.

### 사용 조건

- 질문 유형이 뚜렷하게 갈린다.
- 유형별로 답변 방식, 도구, 안전 정책, 출력 형식이 달라야 한다.
- 모든 질문을 하나의 프롬프트로 처리하면 품질이 흔들린다.

### 추천 라우팅 기준

| 질문 유형 | 처리 방식 |
| --- | --- |
| 단순 설명, 정의, 비교 | Augmented LLM 또는 일반 단일 답변 |
| 최신 뉴스, URL, 문서 기반 질문 | Augmented LLM |
| 순차 작업 | Prompt chaining |
| 여러 독립 관점이 필요한 분석 | Parallelization |
| 예측 불가능한 하위 작업이 많은 복합 작업 | Orchestrator-workers |
| 정책, 안전, 법률, 의료, 금융처럼 고위험 질문 | Routing 후 전문 안전 프롬프트 적용 |

### 답변 패턴

```text
1. 질문을 분류한다.
2. 분류 결과에 맞는 답변 프롬프트를 선택한다.
3. 선택한 프롬프트로 답변한다.
4. 하단에 최종 답변 방식과 라우팅 결과를 표시한다.
```

### 하단 표시 예

```text
---
답변 방식: Routing
선택 이유: 질문 유형을 먼저 분류한 뒤 뉴스 분석용 답변 경로로 처리했습니다.
```

## 4. Parallelization

### 정의

서로 독립적인 하위 작업을 동시에 처리한 뒤 결과를 합친다. 같은 질문을 여러 관점으로 답하게 해 투표하거나, 섹션별로 나눠 처리할 수도 있다.

### 사용 조건

- 하위 작업들이 서로 의존하지 않는다.
- 속도를 높이거나 다중 관점 검토가 필요하다.
- 보안, 사실성, 품질 평가처럼 별도 검토 축이 필요하다.

### 예시

- "50개 기사에서 경제, 기술, 정책 관점을 각각 분석해줘"
- "동일한 기사 목록을 긍정/부정/리스크 관점으로 나눠 평가해줘"
- "답변 생성과 안전성 검토를 동시에 수행해줘"

### 답변 패턴

```text
1. 독립 가능한 하위 작업을 나눈다.
2. 각 작업을 병렬로 실행한다.
3. 결과를 병합한다.
4. 충돌하는 결론이 있으면 명시한다.
5. 하단에 답변 방식을 표시한다.
```

### 하단 표시 예

```text
---
답변 방식: Parallelization
선택 이유: 기술, 시장, 정책 관점을 독립적으로 분석한 뒤 종합했습니다.
```

## 5. Orchestrator-workers

### 정의

중앙 오케스트레이터가 질문을 보고 필요한 하위 작업을 동적으로 정하고, 여러 worker에게 맡긴 뒤 최종 답변을 합성한다.

### 사용 조건

- 어떤 하위 작업이 필요한지 사전에 고정하기 어렵다.
- 여러 자료원 탐색, 코드 수정, 복합 리서치처럼 작업 범위가 입력마다 크게 달라진다.
- 중간에 추가 검색, 재시도, 검증, 보완이 필요할 수 있다.

### 예시

- "이 주제에 대해 여러 출처를 찾아 종합 리포트를 만들어줘"
- "뉴스 크롤러를 개선하고, 실패 원인을 찾아 고치고, 테스트해줘"
- "경쟁사 5곳을 조사해 기능, 가격, 리스크를 비교해줘"

### 답변 패턴

```text
1. 오케스트레이터가 사용자 목표와 성공 기준을 정한다.
2. 필요한 하위 작업을 동적으로 만든다.
3. 각 worker가 자료 수집, 분석, 검증, 작성 등을 수행한다.
4. 오케스트레이터가 결과를 비교하고 충돌을 정리한다.
5. 최종 답변을 합성한다.
6. 하단에 답변 방식을 표시한다.
```

### 하단 표시 예

```text
---
답변 방식: Orchestrator-workers
선택 이유: 필요한 하위 조사 항목을 사전에 고정하기 어려워 작업을 동적으로 분해했습니다.
```

## 별도 패턴

아래 경우에는 5개 패턴에 억지로 끼워 넣지 않고 별도 패턴으로 표시한다.

### Direct answer

간단한 사실, 정의, 코드 한 줄 설명처럼 외부 도구나 단계 분해가 필요 없는 경우.

```text
---
답변 방식: 별도 패턴: Direct answer
선택 이유: 추가 도구나 다단계 처리가 필요 없는 단순 질의였습니다.
```

### Clarification-first

요구사항이 모호하거나 잘못 해석하면 위험한 경우. 먼저 짧게 확인 질문을 한다.

```text
---
답변 방식: 별도 패턴: Clarification-first
선택 이유: 필수 조건이 부족해 바로 답변하면 잘못된 결과가 나올 수 있습니다.
```

### Evaluator-optimizer

Anthropic 글에는 포함되어 있지만 사용자가 지정한 5개 목록에는 없다. 명확한 평가 기준이 있고 반복 개선으로 품질이 좋아지는 번역, 문서 다듬기, 코드 리뷰, 리포트 품질 개선에 적합하다.

```text
---
답변 방식: 별도 패턴: Evaluator-optimizer
선택 이유: 초안 생성 후 평가와 개선 루프가 필요한 작업입니다.
```

## 라우터용 시스템 프롬프트 예시

```text
너는 사용자 질문을 답변하기 전에 적절한 워크플로 타입을 선택한다.

선택 가능한 타입:
1. Augmented LLM
2. Prompt chaining
3. Routing
4. Parallelization
5. Orchestrator-workers

위 타입에 맞지 않으면 "별도 패턴: <패턴명>"을 사용한다.

선택 기준:
- 외부 문서, 검색 결과, 크롤링 결과를 붙여 단일 답변이면 Augmented LLM
- 고정된 순차 단계가 필요하면 Prompt chaining
- 질문 유형 분류 후 전문 처리 경로가 필요하면 Routing
- 독립 하위 작업을 동시에 처리하거나 여러 관점 검토가 필요하면 Parallelization
- 하위 작업을 사전에 알 수 없고 동적으로 분해해야 하면 Orchestrator-workers
- 단순 질의는 별도 패턴: Direct answer
- 명확한 평가 기준으로 반복 개선이 필요하면 별도 패턴: Evaluator-optimizer
- 질문이 모호하면 별도 패턴: Clarification-first

답변 마지막에는 반드시 아래 형식을 붙인다.

---
답변 방식: <선택한 타입>
선택 이유: <한 문장>
```

## 기술적으로 명확히 지시해야 하는 지점

1. 분류 결과를 구조화된 값으로 받기

LLM이 자유 문장으로만 답하면 후처리가 어렵다. 먼저 JSON으로 `workflow_type`, `reason`, `needs_tools`, `confidence`를 받도록 지시한다.

2. 워크플로 타입 목록 고정

타입명을 임의로 바꾸지 못하게 해야 한다. 예를 들어 `Paralleization`처럼 오타가 있는 입력도 내부적으로는 `Parallelization`으로 정규화한다.

3. 도구 사용 가능 범위 명시

검색, 크롤링, 파일 읽기, DB 조회, 계산기 등 어떤 도구를 쓸 수 있는지 LLM에게 정확히 알려야 한다. 도구 설명이 모호하면 잘못된 경로를 선택한다.

4. 타임아웃과 최대 반복 횟수 설정

Prompt chaining, Orchestrator-workers, Evaluator-optimizer는 무한히 길어질 수 있다. 전체 제한 시간, 최대 단계 수, 최대 검색 횟수, 최대 worker 수를 코드 변수로 분리해야 한다.

5. 중간 결과 검증 기준 정의

뉴스 크롤링이라면 URL 존재 여부, 제목 공백 여부, 중복 제거 기준, 출처 신뢰도, 최신성 기준을 코드와 프롬프트에 함께 정의해야 한다.

6. 하단 표시 강제

최종 답변 생성 프롬프트에 `답변 방식` 블록을 필수 출력 형식으로 넣고, 서버 코드에서도 없으면 자동으로 붙이는 보정 로직을 두는 것이 좋다.

## 현재 대시보드 구현 매핑

현재 Streamlit 대시보드는 `src/workflow_patterns.py`에서 워크플로 타입을 결정하고,
`dashboard.py`의 답변 하단에 아래 형식으로 표시한다.

```text
답변 방식: <workflow type>
선택 이유: <reason>
```

### 적용 규칙

| 조건 | 표시 타입 | 실제 구현 |
| --- | --- | --- |
| 담당을 선택하지 않고 질문 | Routing | 질문 키워드를 기준으로 담당 Agent를 자동 배정한 뒤 답변 |
| 담당 1명을 직접 선택 | Augmented LLM | 선택 담당 프롬프트 + 집계 대시보드 컨텍스트로 단일 답변 |
| 담당 2명 이상 선택 | Parallelization | 담당별 독립 답변을 생성하고 채팅에 연속 표시 |
| 순차 처리 의도가 강한 질문 | 별도 패턴: Workflow intent detected | 현재는 실제 Prompt chaining을 실행하지 않고 한 답변으로 처리 |
| 동적 작업 분해 의도가 강한 질문 | 별도 패턴: Orchestration intent detected | 현재는 실제 Orchestrator-workers를 실행하지 않음 |
| 평가·개선 루프 의도가 강한 질문 | 별도 패턴: Evaluator-optimizer | 사용자가 지정한 5개 외 Anthropic 추가 패턴으로 표시 |

### 현재 기술적 한계

- `Prompt chaining`은 아직 실제 다중 LLM 호출, 중간 산출물, gate 검증을 수행하지 않는다.
- `Parallelization`은 UI/논리상 다중 담당 답변이지만 Streamlit 렌더링 안정성을 위해
  실제 비동기 병렬 호출이 아니라 순차 스트리밍으로 표시한다.
- `Orchestrator-workers`는 worker 생성, 작업 한도, 도구 사용 권한, 합성 규칙이 아직
  코드로 정의되어 있지 않아 자동 작업 분해를 수행하지 않는다.
- 답변 방식 표시는 LLM 출력에 맡기지 않고 서버 코드에서 강제 표시한다.
